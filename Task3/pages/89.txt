Я Наталья Калачева, Frontend-разработчик в <a href="https://agima.ru/?utm_source=habr&amp;utm_medium=article&amp;utm_campaign=kalacheva_front_2">AGIMA</a>. Эта статья посвящена правилам, которые помогают упростить поддержку и расширение приложений на Vue. Тут я рассказываю, как организовать хранение компонентов, стилей и плагинов, когда использовать стор и полезные функции Vue. <a href="https://habr.com/ru/company/agima/blog/722332/"><u>Первые 3 принципа я опубликовала вчера.</u></a> Здесь еще 3.</p><figure class="full-width "><img src="https://habrastorage.org/r/w780q1/getpro/habr/upload_files/48b/33c/eda/48b33ceda5511849430a1d23fd49e9a9.jpg" width="2400" height="1200" data-src="https://habrastorage.org/getpro/habr/upload_files/48b/33c/eda/48b33ceda5511849430a1d23fd49e9a9.jpg" data-blurred="true"/></figure><h2>4. Оборачивать библиотеки при инициализации</h2><p>Лучше всего создавать отдельный файл для инициализации широко используемых плагинов. Например, вместо того чтобы тянуть Axios в компонентах непосредственно из зависимостей, вы можете создать класс с более общим именем — такой, как HTTP, чьи методы вызывают Axios под капотом. Например:</p><pre><code class="javascript">import axios from 'axios';
class $Http {
 constructor(options) {
   this.instance = axios.create({
     baseURL: options?.baseURL ?? '',
     headers: options?.headers ?? {},
     params: options?.params ?? {},
   });
 }
 get = async (resource, params) => this.instance.get(resource, { params });
 // другие методы класса, интерцепторы и т.д.
}
export default $Http;</code></pre><p>На первый взгляд, это бесполезный модуль, ведь он делает то же самое, что делал бы модуль из зависимости. Только вы вместо этого пишете импорт HTTP из «http.js». Однако это облегчает переиспользование кода и расширение общей функциональности. Мы можем расширять наш класс и добавлять новые функции/конфиги глобально.</p><p>Например, с Axios это может быть использование интерсепторов. Мы можем всегда предупреждать пользователя, когда запрос <strong>ajax</strong> терпит неудачу, а не использовать <strong>catch</strong> везде, где делаем запрос.</p><p>Если же мы решим перейти на другой плагин или обновиться до версии с критическими изменениями, не нужно рыскать по всей кодовой базе в поиске всех мест, где это использовалось. Достаточно адаптировать наш класс к изменениям.</p><p>Конечно, не все библиотеки стоит оборачивать. Если есть сомнения, можно задать себе ряд вопросов:</p><ul><li><p><strong>Вы не уверены в том, что реализация библиотеки на 100% удовлетворяет будущие задачи?</strong></p></li><li><p><strong>Эта зависимость требует настройки и расширения функционала?</strong></p></li><li><p><strong>Будете ли это переиспользовать в разных частях приложения?</strong></p></li><li><p><strong>Вы будете использовать только часть функционала?</strong></p></li></ul><p>Если я могу с уверенностью ответить «да» на часть этих вопросов, значит, дополнительная работа над оберткой оправдана.</p><h2>5. Валидировать пропсы, использовать типы</h2><p>Во время разработки большого проекта вы можете забыть точный тип и формат данных, которые вы передавали в компонент. Поэтому важно всегда описывать тип данных. Это делает код более читабельным и помогает подсветить ошибки на этапе разработки. Vue предоставляет мощный встроенный механизм для проверки этих данных. Это гарантирует, что компонент будет использоваться по назначению.</p><pre><code class="javascript">
export default {
 name: 'BlockName',
 props: {
   title: {
     type: String,
     default: ''
   }
 }
}
</code></pre><p>Если вы используете Typescript, то сложные типы можно описывать с помощью интерфейсов:</p><pre><code class="typescript">import { PropType, defineComponent } from 'vue'


interface IItem {
 id: string
 title: string
}


export default defineComponent({
 name: 'BaseList',
 props: {
   list: {
     type: Array as PropType&lt;IItem[]>,
     default: () => []
   }
 }
})
</code></pre><p>Лучший способ ограничить пропсы определенным набором значений — использовать параметр валидатора. Это функция проверки, которая принимает значение и возвращает один из двух результатов — true или false.</p><pre><code class="typescript">export default {
 name: 'BaseIcon',
 props: {
   position: {
     type: String,
     validator: s => ['top', 'left'].includes(s)
   }
 }
};
</code></pre><p>Всё вышеперечисленное справедливо при использовании Vue 3 либо с <strong>options</strong>, либо с <strong>composition API</strong>. Разница только в <strong>setup</strong>: пропсы должны быть объявлены с помощью функции <strong>defineProps()</strong>. Например:</p><pre><code class="typescript">&lt;script setup lang="ts">
interface IProps {
 title: string
 list: Array&lt;IItem>
}


const { title, list } = defineProps&lt;IProps>()
&lt;/script></code></pre><p>В сочетании с TypeScript встроенные механизмы валидации и типизирования пропсов могут обеспечить правильное использование компонента, уменьшить количество ошибок и упростить общее понимание кода.</p><h2>6. Организовать стили</h2><p>Даже если логика нашего приложения написана идеально, CSS может значительно подпортить внешний вид и создать множество проблем при дебаге и расширении/изменении дизайна. Важно уделять стилям повышенное внимание, чтобы избежать конфликтов и непредсказуемого поведения.</p><p>Есть множество методологий написания CSS, например БЭМ, OOCSS, SMACSS, Atomic CSS и т. п. Среди этих подходов нет идеального — у всех свои плюсы и минусы. Поэтому можно выбрать один, а можно совместить несколько. <strong>Важно договориться с командой о едином подходе</strong>. Тогда масштабирование проекта будет даваться легче.</p><p><strong>Нужно прописать глобальные стили, которые распространяются на всё приложение.</strong> Есть несколько стилей, которые мы хотим применить ко всем компонентам. Например, normalize.css или reset.css. Или общие стили, такие как шрифты, размеры заголовка, цвета и CSS-переменные. Добавим их в тег стиля App.vue</p><pre><code class="javascript">&lt;template>
 &lt;div id="app">
   &lt;main>
     &lt;router-view>&lt;/router-view>
   &lt;/main>
 &lt;/div>
&lt;/template>


&lt;style>
body {
 margin: 0;
}
&lt;/style></code></pre><p><strong>Стили компонента всегда изолируются при помощи директивы scoped. </strong>Это гарантирует, что любой стиль CSS, определенный в компоненте, будет применяться только к этому шаблону и не создаст конфликта стилей за его пределами.</p><pre><code class="css">&lt;style lang="scss" scoped>
.Item {
 &amp;__text {
   margin-bottom: 30px;
 }
}
&lt;/style>
</code></pre><p>Если стилей одного компонента много или одни и те же стили переиспользуются в нескольких компонентах, в <strong>assets</strong> я создаю отдельную папку <strong>styles/components</strong> и добавляю туда стили, которые в дальнейшем импортирую в компонент.</p><pre><code class="css">&lt;style lang="scss" scoped>
@import "@/assets/styles/components/table.scss";
&lt;/style></code></pre><p>Однако использование <strong>scoped</strong> не отменяет использование классов для стилизации элементов. Это наиболее производительный вариант селектора по сравнению с названием тегов. <a href="https://vuejs.org/api/sfc-css-features.html#scoped-css"><u>Подробнее об этом в документации</u></a>.<br/><br/><strong>Как альтернативу можно использовать CSS Module.</strong> Они также решают проблему конфликта стилей. Но модули предполагают динамический рендеринг и отображение имен классов (например, можно настроить классы в зависимости от среды разработки или props).</p><pre><code class="javascript">&lt;template> 
 &lt;section>
   &lt;div>
     &lt;h1 :class="$style.heading">Заголовок&lt;/h1>
   &lt;/div>
 &lt;/section>
&lt;/template>




&lt;style module>
.heading {
 font-size: 50px;
 font-weight: 900;
 text-align: center;
}
&lt;/style>
</code></pre><p><br/>Данный модуль генерирует следующий CSS:</p><pre><code class="css">.ComponentName__heading__2Kxy { 
 font-size: 50px;
 font-weight: 900;
 text-align: center;
}
</code></pre><h2>Заключение</h2><p>Эта статья дает краткое описание хороших практик при разработке современных клиентских приложений на Vue. Тем не менее, нет правильного или неправильного ответа. Ни один подход определенно не лучше других. Это полностью зависит от ситуации и предпочтений разработчика.</p><p>Есть много хороших практик, которые не перечислены в этой статье.
